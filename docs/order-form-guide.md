# Форма заказа букета — краткий обзор

## Что сделано

Ветка `feat/order-form`. Реализованы три связанных куска:

1. **Рабочая форма заказа букета** — `/order/<slug>/`. Раньше view только рендерил пустой шаблон, теперь принимает данные, сохраняет `Order` в БД и редиректит на страницу «спасибо».
2. **Страница благодарности после заказа** — `/order/done/` (`order_done.html`).
3. **Страница благодарности после заявки на консультацию** — `/consultation/done/` (`consultation_done.html`). Это «окно после отправки заявки» с доски. Саму форму консультации оживляет Максим — когда он закончит, ему останется сделать `return redirect('consultation-done')`, и экран подхватится.

## Какие файлы изменил/создал

| Файл | Что сделал |
|------|-----------|
| `app/forms.py` (новый) | `OrderForm` — Django `ModelForm` на модель `Order`, 4 поля: имя, телефон, адрес, время доставки |
| `app/urls.py` | Заменил `path('order/', …)` на `path('order/<slug:slug>/', …)`. Добавил `order/done/` и `consultation/done/` |
| `app/views.py` | Переписал `order(request, slug)` — GET/POST. Добавил `order_done`, `consultation_done`. Заодно причесал импорты (было два раза `from django.shortcuts import`) |
| `templates/order.html` | Оживил форму — `method="post"`, `{% csrf_token %}`, поля формы через `{{ form.XXX }}`, радиокнопки через `{% for radio in form.delivery_time %}`. Вверху — плашка с букетом (картинка + название + цена). Кнопка «Оплатить» → «Подтвердить заказ». Кнопка «Назад» — ссылка на карточку букета |
| `templates/card.html` | Кнопка «Заказать букет» теперь передаёт slug: `{% url 'order' bouquet.slug %}` |
| `templates/order_done.html` (новый) | Thank-you страница после заказа |
| `templates/consultation_done.html` (новый) | Thank-you страница после консультации |

## Почему так, а не иначе

### 1. Зачем `ModelForm`, а не ручной `forms.Form`?

Модель `Order` уже есть (написал Максим). Всё, что форма должна сохранить — это 4 из её полей. `ModelForm` автоматически:
- делает поля из модели (включая choices `delivery_time`),
- валидирует типы и длины,
- сохраняет объект через `form.save()`.

Ручной `forms.Form` заставил бы дублировать список полей и руками создавать `Order(...)` в view. `ModelForm` — стандартный Django-путь для таких форм.

### 2. Почему URL с `<slug>`, а не передача через сессию?

Альтернативы были такие:
- **Через сессию**: на кнопке «Заказать» сохраняем `bouquet_id` в `request.session`, а на форме заказа его читаем.
- **Через GET-параметр**: `?bouquet=svadebnyj`.
- **Через slug в URL**: `/order/svadebnyj/` — выбрал этот.

Плюсы slug в URL:
- букет виден в адресной строке — понятно что заказывается,
- работает по прямой ссылке (можно кинуть в закладки, поделиться),
- идиоматично для REST: ресурс в пути,
- не надо «греть» сессию до формы.

Минус один — если зайти на `/order/` без slug, это 404. Но в реальном сценарии попасть туда можно только кликом с карточки букета, а в карточке слаг всегда есть.

### 3. Почему `form.save(commit=False)` → поле → `save()`?

`ModelForm.save()` сохраняет объект в БД. Но в форме у нас нет поля `bouquet` — букет берётся из URL. Если вызвать `form.save()` сразу, Django попробует сохранить `Order` без заполненного FK `bouquet` и упадёт.

`commit=False` возвращает собранный объект в памяти, но в БД не пишет. Ставлю букет и дожимаю `.save()`:

```python
new_order = form.save(commit=False)
new_order.bouquet = bouquet
new_order.save()
```

Это стандартный приём для ModelForm с полями, которые приходят не из формы, а из контекста запроса.

### 4. Почему `delivery_time` рендерится через `{% for radio in form.delivery_time %}`?

Django-формы по умолчанию рендерят `RadioSelect` как `<ul>` с либами. Нам нужна та же разметка, что в макете — `<div class="order__form_radioBlock">` с отдельным `input` и `label`.

Итерация `form.delivery_time` отдаёт каждый radio-option как объект с `tag` (сам input) и `id_for_label`, `choice_label` (подпись). Получается полный контроль над HTML-обёрткой, стили из макета работают без правок.

### 5. Почему разделил thank-you страницы на две (`order_done` и `consultation_done`)?

Формально можно было сделать одну универсальную страницу с переменным текстом. Но это даёт разное UX:
- После заказа — «флорист свяжется по доставке».
- После консультации — «перезвонит в течение 20 минут, посмотрите коллекцию».

Разные тексты, разный смысл. Проще держать двумя шаблонами, чем городить условную логику на одной странице.

### 6. Почему не трогал оплату (`order-step.html`)?

Оплата в ТЗ — «возможно будет круто, если популярно», на доске отдельная карточка «Оплата Юмани» в очереди. В MVP заказа нет — есть сохранение в БД + флорист/курьер обрабатывает вручную. Шаблон `order-step.html` остался нетронутым, он пригодится, когда кто-то возьмёт задачу про ЮMoney.

## Как работает POST, шаг за шагом

1. Пользователь на карточке `/bouquet/svadebnyj/` жмёт «Заказать букет».
2. Браузер открывает `GET /order/svadebnyj/`. View `order(request, slug)` получает slug, находит `Bouquet`, создаёт пустой `OrderForm()` и рендерит `order.html`.
3. Пользователь заполняет форму и жмёт «Подтвердить заказ». Браузер шлёт `POST /order/svadebnyj/` с данными + CSRF.
4. View снова получает slug, снова достаёт `Bouquet`, но теперь создаёт `OrderForm(request.POST)`.
5. Если `form.is_valid()` — создаётся объект `Order` через `save(commit=False)`, ему дописывается `bouquet`, сохраняется в БД. Дальше `redirect('order-done')`.
6. Если форма невалидна — `order.html` рендерится заново, уже с ошибками в `form.XXX.errors`.
7. На редиректе браузер делает `GET /order/done/`, показывается «Заказ принят».

## Зависимости и связанные штуки

- Модель `Order` не менялась — работает с полями, которые уже были у Максима.
- Миграций новых нет (форма — это про логику, не про схему БД).
- Ветка `feat/bouquet-matching` (логика подбора) независимо идёт — после её мёржа в main **надо будет в отдельном микро-PR обновить `result.html`**, чтобы кнопка «Заказать букет» там тоже передавала `bouquet.slug`. Сейчас в этой ветке ей заняться нельзя (конфликт с другой веткой).

## Важная мелочь про порядок путей в `urls.py`

`path('order/<slug:slug>/', ...)` должен быть **после** `path('order/done/', ...)`. Django резолвит пути сверху вниз и первый совпадающий выигрывает. Если slug-вариант стоит раньше, `/order/done/` заматчится как slug=`done`, view полезет в БД искать букет с таким slug'ом и отдаст 404. У меня сразу был этот баг — переставил местами и всё стало ок.

## Мини-план тестирования перед мёржем

1. `/catalog/` → клик букета → карточка → «Заказать букет» → форма открывается с правильным букетом.
2. Отправить валидную форму → страница «Заказ принят» → в админке появилась запись `Order` со статусом `Новый`.
3. Отправить форму с пустым телефоном → форма возвращается с ошибкой, поле выделено.
4. Нажать «Назад» на форме → вернулся на карточку букета.
5. Открыть `/consultation/done/` напрямую — страница отрисовывается.
